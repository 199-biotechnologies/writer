## Benchmark Strategy (user decision from this session)

The autoresearch loop optimises toward a **weighted combined score** of three benchmarks. These were picked because they measure exactly what writer is selling — voice fidelity, AI-tell avoidance, and baseline prose quality — not irrelevant things like math reasoning or code generation.

### Benchmark 1: VoiceFidelity (PRIMARY — weight 0.50)

**What it measures:** How close is a generated text to the user's own distribution?

**How:** Hold out 10% of the user's samples during ingestion. For each held-out sample, generate a rewrite via writer (`writer rewrite <sample>`). Compute the stylometric distance between the generated output and the held-out sample using `stylometry::scoring::distance` (Phase 2 delivers this function). Lower is better.

**Metric:** Mean stylometric distance across held-out samples, range 0.0-1.0.

**Target:**
- Baseline (vanilla Gemma 4 26B-A4B, no LoRA, no decoding layer): ~0.55-0.65 expected
- Phase 4 target (decoding layer only, no training): ≤ 0.40
- Phase 5 target (SFT LoRA): ≤ 0.30
- Phase 6 target (SFT + DPO): ≤ 0.20
- Phase 7 target (SFT + DPO + contrastive decoding): ≤ 0.15

**Why primary:** This is the product. If this metric doesn't go down, nothing else matters.

**Implementation:** `src/stylometry/scoring.rs` (defined in Phase 2). Benchmark runner in `src/bench/voice_fidelity.rs` (new file, defined in Phase 3.5 below).

### Benchmark 2: SlopScore (weight 0.30)

**What it measures:** Does the generated output still smell like an LLM, independently of voice?

**How:** Run the existing `humanise-text` detector (`~/.claude/skills/humanise-text/scripts/detect_ai_patterns.py`) on generated outputs. Parse the JSON report. Primary signal: `ai_density_score` (0-100). Secondary: flagged word count, structural pattern count, rhythm SD.

**Metric:** `ai_density_score`, lower is better.

**Target:**
- Baseline: 45-60 expected (generic Gemma 4 output)
- Phase 4 target (with banned-word logit bias): ≤ 30
- Phase 6 target (with DPO training against AI rewrites): ≤ 18
- Phase 7 target (with contrastive decoding): ≤ 12
- Stretch: below 10 (human writing typically scores 5-15)

**Why secondary:** A voice can match the user well while still using AI tics (em-dashes, "Moreover," rule-of-three). SlopScore catches that failure mode.

**Implementation:** Shell out to `python3 ~/.claude/skills/humanise-text/scripts/detect_ai_patterns.py` from the benchmark runner. Parse JSON, extract score.

### Benchmark 3: CreativeWritingRubric (weight 0.20)

**What it measures:** Is the output actually good prose, by standards external to writer's own stylometry?

**How:** Use a small subset of EQ-Bench Creative Writing v3 prompts (10-15 prompts from https://eqbench.com/creative_writing.html — the repo is at https://github.com/EQ-bench/creative-writing-bench). Generate responses via writer. Judge with Claude Sonnet 4.6 (via Anthropic API, user already has this configured) using EQ-Bench's published rubric.

**Metric:** Mean rubric score, 0-100.

**Target:**
- Baseline (Gemma 4 26B-A4B): 70-75 expected
- Don't REGRESS below baseline — this benchmark's job is to catch the case where over-optimising for voice fidelity destroys prose quality

**Why this one instead of a full EQ-Bench run:** Full EQ-Bench takes hours per run; the autoresearch loop needs fast feedback. A subset runs in minutes and still catches major quality regressions. Optionally run the full suite nightly.

**Implementation:** `src/bench/creative_rubric.rs`. Loader for EQ-Bench prompt subset, judge call via `anthropic-sdk-rs` (already in your tooling ecosystem).

### Why NOT the usual suspects

- **MMLU, GSM8K, HumanEval** — measure reasoning and coding, not writing. Irrelevant.
- **MT-Bench** — general chat quality, not voice-specific.
- **Perplexity against held-out corpus** — a model that memorised the corpus would score well but generate poorly on new prompts. Rewards the wrong thing.
- **BLEU / ROUGE / METEOR** — notoriously bad for creative writing; correlate weakly with human judgement.
- **LMArena Creative Writing** — human-judged, not automatable in an autoresearch loop.
- **Writer's own training loss** — not a benchmark; it measures fit to training data, not generalisation to new prompts.

### Combined score formula

```
combined_score = (1.0 - voice_fidelity_distance) * 0.50
               + (1.0 - slop_score / 100.0)       * 0.30
               + (creative_rubric_score / 100.0)  * 0.20
```

All three components are normalised to 0-1 (higher is better) before weighting. The autoresearch loop's single target metric is `combined_score`, to be maximised.

**Baseline expectation (vanilla Gemma 4 26B-A4B, no LoRA, no decoding layer):**
```
combined_score ≈ (1 - 0.60)*0.50 + (1 - 52/100)*0.30 + (72/100)*0.20
              ≈ 0.20          + 0.14          + 0.144
              ≈ 0.484
```

**Phase 7 target:**
```
combined_score ≈ (1 - 0.15)*0.50 + (1 - 12/100)*0.30 + (73/100)*0.20
              ≈ 0.425         + 0.264          + 0.146
              ≈ 0.835
```

**Autoresearch target:** ≥ 0.85 combined score, with no component worse than its baseline by more than 5%.

## Autoresearch Loop Configuration

When Phase 3.5 is done, kick off autoresearch with this target:

```bash
autoresearch start \
  --metric "writer-bench run --json | jq -r '.data.combined_score'" \
  --baseline-command "writer-bench baseline --save" \
  --improvement-threshold 0.01 \
  --max-iterations 50 \
  --eval-timeout 900 \
  --keep-on-regression false
```

**Iteration strategy (per autoresearch skill):**
1. Each iteration modifies one of: decoding parameters (temperature, top_p, n_candidates, logit bias magnitudes, contrastive alpha), training hyperparameters (LoRA rank, alpha, learning rate, max steps), or prompt templates
2. Evaluate via `writer-bench run --json`
3. Keep changes that raise `combined_score` by ≥ 0.01
4. Revert everything else
5. Git commit every keeper with the delta in the message

**Loop cron (separate from autoresearch):**

```bash
loop 2h "writer-bench run --smoke --json | jq -r '.data.combined_score' | tee -a ~/writer-bench-history.log"
```

This is a lightweight drift-tracker so you notice if a dependency bump or Ollama update quietly regresses quality between autoresearch sessions.

## Key Decisions Made (not written in code, would take 45 min to re-derive)

1. **Crate name is `writer-cli` not `writer`** on crates.io. The name `writer` is squatted by an unmaintained 2024 package with 11 recent downloads, no repo link, AI-slop description. Policy-eligible for reclaim but that's slow. Binary name stays `writer` via `[[bin]] name = "writer"`. Brew formula will be named `writer`.

2. **Inference backend = Ollama (shell-out), NOT mistral.rs.** Reversed from an earlier hypothesis. Reasons: Ollama 0.19 moved to Apple MLX as its backend in March 2026, so Ollama-on-Apple-Silicon _is_ MLX; every KV-cache optimisation (TurboQuant, TriAttention) will land in Ollama before it lands in mistral.rs; shelling out keeps writer's code tiny; the trade-off is a soft external dependency but writer's target users are already running Ollama. mistral.rs stays in reserve as a `--features embedded` option for a future single-binary purist mode.

3. **Training backend = mlx-tune (shell-out).** 1006 stars, Unsloth-compatible API, actively maintained, Apple-Silicon native. Writer never writes its own training loop.

4. **LoRA+DPO+contrastive-decoding is the quality stack,** not just LoRA. Pure LoRA underfits; DPO teaches what NOT to do (trains against AI rewrites of the user's own samples); contrastive decoding (CoPe, EMNLP 2025) subtracts the base distribution at decode time. Each layer is worth 10-30% additional voice fidelity improvement; stacked they're the whole product.

5. **Benchmarks are not MMLU/HumanEval/MT-Bench.** The three that matter for writer are VoiceFidelity (custom, stylometric distance to held-out user samples), SlopScore (from humanise-text detect_ai_patterns.py), and a CreativeWritingRubric subset (EQ-Bench v3, 15 prompts, judged by Claude Sonnet 4.6). Combined weighted score targets autoresearch. Full rationale in the Benchmark Strategy section above.

6. **Quality is opt-out, not opt-in.** `writer write` ALWAYS runs the full decoding stack by default. Speed comes from MoE and KV optimisations, not from skipping quality layers. This is codified in the `GenerationParams::default()` and `DecodingConfig::default()` implementations in Phase 0 Tasks 0.4 and 0.9.

7. **The extension point for TurboQuant/TriAttention is structural, not rhetorical.** Concretely: the `KvQuantKind` enum in `src/backends/inference/capabilities.rs` (Task 0.3) already has `TurboQuant`, `TriAttention`, and `Both` variants; the `GenerationParams.kv_quant_preference: Option<KvQuantKind>` field (Task 0.4) passes user preference through; the backend's `capabilities()` method reports what's available; the decoding layer negotiates. Adding TurboQuant once it lands in Ollama is 5 edits in `src/backends/inference/ollama.rs` plus a capability-probe bump. ZERO changes to `src/commands/`, `src/decoding/`, `src/corpus/`, or `src/stylometry/`.

8. **No silent fallbacks ever.** When a capability is missing (backend doesn't support contrastive decoding, adapter missing, Ollama version too old), writer returns `BackendError::CapabilityNotSupported` with a literal remediation suggestion. It never quietly degrades to a worse path.

9. **Data pipeline normalisation is default-on.** `writer learn --clean` is the default; `--raw` is the escape hatch. Raw Twitter exports, Obsidian vaults, email mboxes etc. are always normalised (strip signatures, strip tracking params, dedupe, language-detect) before entering the training corpus. Bad samples destroy a LoRA faster than any architectural choice can fix.

10. **199-biotechnologies org on GitHub is the home, Boris's personal account is NOT.** Global rule per the user's CLAUDE.md. Every `gh` call and every commit uses the org.
